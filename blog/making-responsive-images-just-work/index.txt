<p>Image behavior was spread across scripts and components with repeated conventions:</p>
<ul>
<li>variant naming assumptions,</li>
<li>hardcoded <code>srcSet</code> strings,</li>
<li>multiple script aliases and legacy paths.</li>
</ul>
<p>That made drift easy, and small changes in one place could quietly break expectations elsewhere. I replaced that with a simpler pattern: generate one manifest at build time (<code>src/generated/imageVariantManifest.json</code>) and resolve variants from it at runtime.</p>
<h2>One manifest instead of scattered assumptions</h2>
<p>The main change was replacing repeated conventions with one manifest-driven path. There is now one canonical workflow: <code>yarn image:variants</code>. Rendering logic is less duplicated because a shared <code>ResponsiveImage</code> component replaced repeated <code>&#x3C;picture></code> patterns. Static assets also became data-driven, so background and portrait <code>srcSet</code> values now come from manifest-backed metadata rather than JSX literals.</p>
<h2>A simpler authoring path</h2>
<p>This was an important constraint for me: I do not want to manually create <code>-sm</code>, <code>-md</code>, <code>-lg</code> files every time I add an image.</p>
<p>The workflow is now simple: add the source image, reference it in frontmatter or markdown, then run <code>yarn image:variants</code>. That keeps authoring lightweight while still making outputs consistent.</p>
<h2>Stricter failure modes</h2>
<p>Once maintainability improved, performance work became easier to verify. I also removed silent fallback for required profiles (<code>cover.card</code>, <code>cover.hero</code>, <code>inlineContent</code>). If a required variant is missing, it now fails fast instead of shipping a hidden regression.</p>
<h2>What I actually fixed</h2>
<p>I started by chasing LCP. The durable fix was maintainability: fewer scattered conventions, one source of truth, and clearer build/runtime boundaries.
