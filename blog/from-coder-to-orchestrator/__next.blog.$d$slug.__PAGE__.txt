b:T12fb,<p>Over the past two years, my day-to-day coding workflow has shifted from quick AI help toward clearer task definition and verification. In early 2024, I used AI mostly for autocomplete-level work: snippets, error explanations, and small refactors. By 2026, I still write code when needed, but more of the leverage comes from defining tasks clearly, checking outputs, and fixing edge cases the tools miss.</p>
<p><img src="/assets/blog/from-coder-to-orchestrator/swe-workflow-evolution-content.webp" alt="Timeline diagram comparing software development workflow in 2024 versus 2026" loading="lazy" decoding="async" fetchpriority="low" sizes="(min-width: 1024px) 896px, 100vw" srcset="/assets/blog/from-coder-to-orchestrator/swe-workflow-evolution-content-sm.webp 640w, /assets/blog/from-coder-to-orchestrator/swe-workflow-evolution-content.webp 1024w" width="1024" height="559"></p>
<h2>2024</h2>
<p>I started experimenting with <strong>Cline</strong> for straightforward tasks such as boilerplate, test scaffolding, and repetitive refactors. I avoided giving it larger tasks for two reasons. First, it was easy to get code that looked fine but was wrong in non-obvious ways. Second, anything non-trivial took too much prompt back-and-forth.</p>
<h2>2025</h2>
<p>I switched to <strong>Claude Code</strong> and stopped asking for immediate implementation. Instead, I asked for a plan first and reviewed it.</p>
<p>That improved outcomes, but two problems stayed persistent. Plans were often overbuilt for the actual problem, and the agent would report "done" before handling edge cases. The bottleneck moved from writing code to verification.</p>
<h2>What made the workflow more reliable</h2>
<p>Three changes made the workflow dependable enough for daily use.</p>
<h3>Stronger model reasoning</h3>
<p>For my use cases, newer models have been noticeably better at keeping constraints in context across longer tasks.</p>
<h3>Better repo context</h3>
<p>A lot of bad output came from missing context, not missing capability. I now keep project expectations in <code>CLAUDE.md</code> so every new session starts with the same baseline: architecture preferences, testing requirements, and coding conventions.</p>
<h3>A repeatable execution loop</h3>
<p>The biggest improvement was switching from one-pass execution to a repeatable loop:</p>
<ol>
<li>Plan</li>
<li>Implement</li>
<li>Verify with build/lint/tests</li>
<li>Reflect and continue if needed</li>
</ol>
<p><img src="/assets/blog/from-coder-to-orchestrator/ralph-loop-content.webp" alt="Ralph Loop flow chart showing Plan -> Implement -> Verify -> Reflect" loading="lazy" decoding="async" fetchpriority="low" sizes="(min-width: 1024px) 896px, 100vw" srcset="/assets/blog/from-coder-to-orchestrator/ralph-loop-content-sm.webp 640w, /assets/blog/from-coder-to-orchestrator/ralph-loop-content.webp 1024w" width="1024" height="547"></p>
<h2>Current workflow</h2>
<p>I often split the work into planning and implementation passes. In practice, that means I write a short feature brief with constraints and non-goals, turn that into checkpoints, let the implementation run through the loop per checkpoint, and then review design and risk rather than just syntax.</p>
<p>Example brief:</p>
<blockquote>
<p>Add Google OAuth using existing <code>AuthService</code>. Store tokens in Redis, not SQL.</p>
</blockquote>
<h2>Trade-offs</h2>
<p>This workflow is productive, but it has clear costs:</p>
<ul>
<li><strong>Cost:</strong> Frequent tool calls and retries add up quickly.</li>
<li><strong>Legacy code friction:</strong> Agents struggle when systems rely on undocumented history.</li>
<li><strong>Personal skill drift:</strong> I type less code directly than I used to.</li>
<li><strong>Attention overhead:</strong> Running multiple agents sounds parallel, but review and coordination still funnel through one person. Human attention is limited, and I still don't have a great system for managing that bottleneck consistently.</li>
</ul>
<h2>Where the work moved</h2>
<p>A larger share of the work now sits in requirements, constraints, and review discipline.</p>
<p>Implementation still matters, but I now spend more attention on deciding what to build, writing clearer constraints, and checking whether the result is right.</p>
